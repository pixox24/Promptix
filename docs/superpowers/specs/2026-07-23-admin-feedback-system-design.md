# 后台反馈系统统一设计

## 目标

后台所有操作反馈使用同一套明确、可访问、可预测的规则，避免成功消息固定堆在页面左上角，也避免根据中文文案猜测消息类型。

## 反馈分层

1. 短暂操作结果使用全局 Toast：保存、提交审批、删除、切换状态、生成任务创建等。
2. 当前表单必须处理的问题使用 InlineAlert：字段校验、文件格式、版本冲突、缺少前置条件等。
3. 持续异步状态保留在业务卡片或状态区域：queued、running、审批中、执行中等。
4. 破坏性操作先使用 ConfirmDialog，完成后再使用 Toast。

## 视觉与交互

- Toast 固定在右下角，移动端居中靠下，最多显示 3 条。
- 同文案通知去重；成功、信息、警告、错误分别停留 3、4、6、8 秒。
- 错误通知使用 `role="alert"`，其他通知使用 `role="status"`。
- InlineAlert 的颜色和语义由显式 `type` 决定，禁止通过文案中的“已”“成功”等字样推断。
- 页面加载失败和需要用户修正的问题留在上下文附近，不使用稍后自动消失的 Toast。

## 迁移范围

- 模板编辑和发布
- 智能入库及草稿审阅
- 分类管理
- 模板治理审批、规则和运行中心
- Provider 与 Model 管理

## 验收标准

- “发布请求已提交审批”显示为全局 Toast，不再占据页面左上角。
- 后台不存在基于消息文本推断成功/失败类型的逻辑。
- 所有需要持续显示的错误使用统一 InlineAlert。
- Toast 和 InlineAlert 均具备屏幕阅读器可识别的实时区域语义。

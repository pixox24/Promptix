# 搜索框中文输入稳定性设计

## 1. 背景与问题

首页和模板库的搜索框当前直接以 URL 查询参数 `q` 作为受控输入值。每次 `onChange` 都调用 `setSearchParams`，触发路由状态更新，再由 URL 反向设置输入框 `value`。

普通英文和已经提交的完整中文可以保持一致，但中文输入法在拼音组合、候选词选择阶段会产生未提交文本。此时逐键更新 URL 和路由重渲染可能中断输入法组合，表现为拼音字母残留、字符顺序混乱，或显示内容与用户刚输入的内容不一致。

## 2. 目标

- 中文输入法组合期间，输入框显示值完全由本地草稿状态控制。
- 输入法组合期间不更新 URL、不发起搜索请求。
- 组合结束后提交完整文本。
- 普通英文、数字、粘贴和删除仍使用 300ms 防抖提交。
- 浏览器前进、后退、直接访问带 `q` 的链接时，输入框能够同步 URL。
- 保留现有排序规则：从空搜索进入有效搜索时默认“最相关”；继续输入不得覆盖用户显式选择的排序。

## 3. 方案比较

### 方案 A：只增加 `compositionstart` / `compositionend`

输入框仍直接读取 URL，仅在组合期间阻止写入 URL。改动较小，但组合期间 URL 中的旧值仍可能通过其他路由更新覆盖正在输入的文本，不能彻底隔离显示状态。

### 方案 B：本地草稿状态 + 组合保护 + 防抖提交（采用）

搜索框维护独立的 `draftQuery`，输入事件立即更新草稿；非组合状态下防抖提交，组合结束时再进入同一提交链路。URL 只保存已经提交的查询。

该方案清晰地区分“用户正在编辑的值”和“应用已经执行的查询”，能够同时解决输入法稳定性与逐键路由更新问题。

### 方案 C：搜索按钮手动提交

完全避免输入期间更新 URL，但改变当前即时搜索体验，增加操作成本，不采用。

## 4. 组件与数据流

新增一个聚焦搜索输入行为的组件或 Hook，职责如下：

1. 接收 URL 已提交值 `query` 和提交回调 `onQueryChange`。
2. 使用本地 `draftQuery` 驱动输入框 `value`。
3. `onChange` 始终只同步更新 `draftQuery`。
4. `compositionstart` 将组合状态设为 `true`。
5. 组合期间不执行防抖提交。
6. `compositionend` 将最终文本写入草稿，并交给正常的防抖提交逻辑。
7. 非组合输入停止 300ms 后调用 `onQueryChange(draftQuery)`。
8. 当 URL `query` 因前进、后退或外部导航发生变化，且当前没有输入法组合时，同步到 `draftQuery`。
9. 如果草稿与已提交值相同，不重复提交。

数据流为：

```text
键盘/输入法 → draftQuery → 搜索框显示
                         ↓（非组合且防抖结束）
                    URL q 参数 → API 查询
```

## 5. 边界行为

- 清空输入：草稿立即显示为空，300ms 后删除 URL 中的 `q`，恢复无搜索默认排序。
- 快速连续输入：旧定时器被取消，只提交最后一次稳定值。
- 中文候选阶段：允许拼音和候选文本在输入框内部变化，不触发 URL 导航。
- 粘贴中文或英文：按普通输入处理，300ms 后提交。
- URL 导航：没有组合输入时以 URL 为准；组合期间延迟外部同步，避免破坏输入法会话。
- 组件卸载：清除待执行定时器，避免卸载后提交。
- 桌面侧栏和移动筛选抽屉共用同一输入实现，防止行为漂移。

## 6. 测试设计

新增回归测试覆盖：

1. 普通输入立即更新草稿，但不会立即提交。
2. 300ms 后只提交最后一个草稿值。
3. `compositionstart` 后多次变更不会提交。
4. `compositionend` 后提交最终中文文本。
5. 外部 `query` 变化能够同步草稿。
6. 草稿等于 URL 值时不重复提交。
7. 清空输入删除查询并恢复现有排序语义。
8. 首页和模板库的桌面、移动搜索入口共用组合事件处理。

验证范围包括针对性回归测试、完整测试、Lint 和生产构建。

## 7. 完成条件

- 使用中文输入法连续输入、选词时不出现拼音残留或字符错乱。
- 输入框显示内容与用户当前编辑内容一致。
- URL 和搜索请求只接收已经稳定的查询词。
- 英文、粘贴、清空、浏览器前进后退行为正常。
- 搜索相关排序规则不发生回归。

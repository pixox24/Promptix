# Promptix 搜索相关度排序 UX 优化设计

## 1. 文档状态

- 日期：2026-07-21
- 状态：已确认，立即实施
- 范围：首页与模板库的排序入口、URL 查询状态和公开模板 API 的排序语义
- 不调整：相关度权重、分类筛选规则、分页规则、数据库结构

## 2. 问题

当前“相关”与热门、精选、高赞、最新一起常驻显示。相关度只有在存在关键词时才有业务含义；没有关键词时，后端把所有模板的相关度记为 0，再按使用次数排序，因此结果实际等同热门。

这会产生三个问题：

1. 用户无法理解“与什么相关”。
2. “相关”和“热门”在空搜索时重复。
3. URL 可残留 `sort=relevance`，清空搜索后界面仍显示“相关”，但数据按热门逻辑排列。

## 3. 方案比较

### 方案 A：删除相关度排序

优点是界面最简单；缺点是搜索结果只能按流行度或时间排列，精确名称、标签和分类匹配无法排在前面。相关度算法已有明确权重，删除会降低搜索质量。

### 方案 B：始终保留，增加说明

优点是改动小；缺点是空搜索下仍然没有实际含义，说明文字不能消除与热门重复的问题。

### 方案 C：仅在搜索场景显示“最相关”

有有效关键词时显示并默认使用“最相关”；没有关键词时隐藏它并默认热门。用户在搜索中仍可切换热门、精选、高赞或最新。

选择方案 C。它保留搜索质量，同时让排序入口与用户当前任务一致。

## 4. 状态定义

“有效关键词”定义为 `query.trim().length > 0`。纯空格不算搜索。

| 事件/URL 状态 | 最终排序 | URL `sort` | “最相关”入口 |
|---|---|---|---|
| 首次打开，无关键词 | 热门 | 省略 | 隐藏 |
| 从空搜索开始输入 | 最相关 | 省略，使用搜索默认值 | 显示并选中 |
| 搜索中选择最新 | 最新 | `latest` | 显示但不选中 |
| 选择最新后继续输入 | 最新 | `latest` | 显示但不选中 |
| 最相关状态下清空搜索 | 热门 | 省略 | 隐藏 |
| 搜索中选择最新后清空 | 最新 | `latest` | 隐藏 |
| 直接访问无关键词的 `?sort=relevance` | 热门 | 自动清理 | 隐藏 |
| 浏览器前进/后退 | 按 URL 重新推导 | 规范化 | 随关键词变化 |

默认排序不写入 URL：无搜索默认为 `hot`，有搜索默认为 `relevance`。这样 URL 更短，也避免默认值在状态切换后变成过期显式值。

## 5. 组件与职责

### 5.1 `useTemplateBrowse`

该 Hook 成为查询状态的唯一规则层：

- 保留输入框原始文本用于显示。
- 使用 trim 后的 `normalizedQuery` 判断是否处于搜索状态并提交 API。
- 导出 `hasQuery`、`setQuery()` 和 `setSort()`。
- `setQuery()` 只在“空搜索 → 有搜索”时切到默认最相关；搜索过程中继续输入不会覆盖用户显式选择的排序。
- “有搜索 → 空搜索”且当前为最相关时恢复热门；其他显式排序继续保留。
- 发现无搜索的 `sort=relevance` 时，通过 replace 清理 URL，并以热门发起请求。
- 切换查询或排序时继续重置页码。

页面不再自行拼装排序默认值，避免首页和模板库规则漂移。

### 5.2 `FilterSidebar` / `MobileFilterBar`

- 将文案从“相关”改为“最相关”。
- 根据 `hasQuery` 生成可见排序项。
- 桌面端无搜索时显示四项，有搜索时显示五项。
- 移动端排序按钮使用两列网格，避免五项横向挤压。
- 移动筛选摘要使用同一可见排序列表查找名称。

### 5.3 首页与模板库

两页直接使用 Hook 暴露的 `setQuery()`、`setSort()` 和 `hasQuery`，不重复实现 URL 默认值逻辑。

### 5.4 公开模板 API

保留现有相关度权重。若客户端显式发送空搜索的 `sort=relevance`，服务端将其规范为 `hot`，保证 API 语义与界面一致。有效关键词为空时默认热门，有效关键词存在时默认最相关。

## 6. 相关度算法（保持不变）

| 匹配 | 分数 |
|---|---:|
| 名称完全匹配 | 100 |
| 自由标签或分类名称完全匹配 | 80 |
| 名称包含 | 60 |
| 摘要包含 | 40 |
| 分类别名完全匹配 | 35 |
| 描述包含 | 20 |
| Prompt 包含 | 5 |

相关度相同时依次按使用次数降序、创建时间降序、模板 ID 升序排列。

## 7. 测试设计

新增 Web 契约测试覆盖：

1. 排序文案为“最相关”。
2. 桌面端和移动端均使用按 `hasQuery` 过滤后的排序列表。
3. Hook 使用 trim 后的关键词判定搜索状态。
4. 空搜索的 `relevance` 被规范为热门并从 URL 清除。
5. 开始搜索默认最相关。
6. 搜索中显式选择其他排序后，继续输入保持该排序。
7. 清空最相关搜索恢复热门。
8. 首页和模板库都使用 Hook 的统一动作。

API 契约测试覆盖空搜索显式 `relevance` 被规范为 `hot`，并确认原相关度权重没有删除。

4. 简化首页和模板库接线。

## 9. 完成条件

- 空搜索时用户看不到“最相关”，默认热门。
- 输入有效关键词后自动出现并选中“最相关”。
- 搜索过程中可稳定切换其他排序，不因继续输入而重置。
- 清空搜索不会留下无意义的 `sort=relevance`。
- 首页、模板库、移动端和直接 API 调用具有一致语义。

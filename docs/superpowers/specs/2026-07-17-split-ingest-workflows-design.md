# 智能入库双流程拆分设计

## 背景

当前 `/admin/ingest` 把“提示词优化”和“图片反推”放在同一表单中，通过文本与文件互斥来决定提交 `text_expand` 或 `image_reverse` 任务。两类任务共用模型选择、消息、任务、轮询和结果状态，用户难以把它们理解为目的不同的独立工作流，也无法在切换任务时保留各自进度。

后端任务类型和 `TemplateDraft` 输出契约已经独立。本设计保留单一“智能入库”页面和现有任务类型，重点拆分前端工作流，并为两条流程增加可管理的全局系统提示词。

## 目标

- 在同一“智能入库”页面提供“提示词优化”和“图片反推”两个明确入口。
- 每条流程独立维护输入、模型、系统提示词、异步任务、轮询状态和结果。
- 两条流程均采用“输入 → 系统提示词 → 结果校对”的三段式结构。
- 管理员可临时修改本次系统提示词，也可将当前内容保存为该流程新的全局默认值。
- AI 生成结果必须经过完整、可编辑的模板草稿校对后才能保存。
- 切换入口不清空状态，也不中断另一条流程正在进行的轮询。

## 非目标

- 系统提示词版本历史或回滚。
- 按 Provider 或 Model 保存系统提示词。
- 任务取消。
- 浏览器刷新后的工作区自动恢复。
- 模板编辑器重构。
- 更改 `text_expand`、`image_reverse` 或 `TemplateDraft` 的核心业务含义。

## 方案比较与选择

设计阶段比较了三种同页布局：

1. 入口卡片加聚焦工作区：入口清晰，进入后只显示当前流程，独立感和专注度最好。
2. 标签页：最紧凑，但两个入口的业务目的不够突出。
3. 双流程并排：适合并行操作，但信息密度高，结果展开后拥挤，小屏体验较差。

采用方案 1。页面顶部固定展示两个入口卡片；用户选择后，下方只显示当前流程的聚焦工作区。入口卡片同时承担后台状态摘要，使用户切换后仍能看到另一条流程的进度。

## 页面与交互设计

### 页面框架

`/admin/ingest` 路由保持不变。页面由以下区域组成：

1. 页面标题和简短说明。
2. 两个入口卡片：提示词优化、图片反推。
3. 当前流程的聚焦工作区。

入口卡片显示流程名称、用途说明和轻量状态：`未开始`、`排队中`、`生成中`、`待校对` 或 `失败`。选择卡片只改变可见工作区，不卸载另一条流程的控制器或停止其轮询。

### 提示词优化流程

1. **输入原始提示词**
   - 输入现有提示词或创意需求。
   - 选择具有 `text` 和 `structured_output` 能力的模型。
2. **系统提示词**
   - 默认折叠，显示“正在使用提示词优化全局预设”。
   - 高级展开后可编辑当前系统提示词。
   - 提供“恢复全局预设”和“保存为系统提示词”。
3. **生成结果**
   - queued/running 期间显示明确状态。
   - succeeded 后显示完整的可编辑 `TemplateDraft`。
   - 校对通过后保存为草稿模板。

### 图片反推流程

1. **上传参考图片**
   - 支持现有允许的图片格式与 10 MB 上限。
   - 显示本地预览，并允许替换图片。
   - 选择最终优化与结构化模型；视觉理解模型继续由现有模型路由自动选择。
2. **系统提示词**
   - 默认折叠，显示“正在使用图片反推全局预设”。
   - 高级展开后可编辑当前系统提示词。
   - 提供“恢复全局预设”和“保存为系统提示词”。
3. **生成结果**
   - 与提示词优化使用同一个草稿校对组件。
   - 校对通过后保存为草稿模板。

图片反推的可配置系统提示词用于最终模板结构化阶段。视觉模型的图片事实描述指令属于内部实现并保持固定，避免运营提示词削弱视觉事实提取。无论主模型直接理解图片，还是先由默认视觉模型描述再交给文本模型结构化，最终结构化阶段都使用本次任务快照中的图片反推系统提示词。

### 系统提示词高级设置

两条流程各自维护一份全局默认值，互不影响。高级面板遵循以下状态规则：

- 初次加载显示服务端全局默认值。
- 本地编辑后显示“仅本次修改”，不会自动写入全局配置。
- “恢复全局预设”用最近一次从服务端获得的全局值覆盖本地编辑内容。
- “保存为系统提示词”将当前内容写为该流程新的全局默认值；成功后，本地内容保持不变并显示“全局预设已更新”。
- 保存全局默认值和提交生成任务是两个独立操作；保存失败不会阻止用户用当前内容提交本次任务。
- 系统提示词去除首尾空白后必须为 1–20,000 个字符。

### 结果校对

两条流程共享 `TemplateDraftReview`，完整开放以下字段：

- 名称、摘要、描述、分类。
- 标签、适用场景。
- 变量列表及每个变量的 key、label、type 和相关选项。
- Prompt 骨架。
- 负面提示词。

校对组件使用共享 `templateDraftSchema` 做客户端校验，并在字段附近显示错误。非法草稿不能保存。保存时调用现有模板创建接口，并写入：

- `source`：对应 `text_expand` 或 `image_reverse`。
- `sourceMeta.jobId`：生成该草稿的任务 ID。

保存成功后直接进入现有模板编辑器，继续处理封面、补充信息和发布。

## 前端组件边界

- `IngestPage`：页面布局、当前入口选择和入口状态摘要。
- `TextOptimizeFlow`：文本输入、模型选择、任务提交和本流程状态。
- `ImageReverseFlow`：图片输入、模型选择、任务提交和本流程状态。
- `SystemPromptPanel`：加载、编辑、恢复和保存单个流程的全局系统提示词。
- `TemplateDraftReview`：共享的完整草稿编辑与校验界面。
- `useIngestJob`：封装任务提交后的轮询、终态处理、临时连接错误与重试。

两个流程控制器在页面生命周期内都保持挂载。`IngestPage` 只控制哪个工作区可见，因此切换入口不会清空状态或停止轮询。每条流程同一时间只允许一个 queued/running 任务；另一条流程仍可独立提交。

## 持久化模型

新增 `ingest_system_prompts` 表：

| 字段 | 说明 |
|---|---|
| `flow_type` | 主键，只允许 `text_expand` 或 `image_reverse` |
| `prompt` | 非空系统提示词，最大 20,000 字符 |
| `updated_by` | 最近修改管理员，可为空以兼容系统初始化 |
| `created_at` | 创建时间 |
| `updated_at` | 最近更新时间 |

迁移创建两条记录。提示词优化默认值和图片反推默认值都保留现有 `TemplateDraft` schema 约束，但分别加入与各自目标相关的指令。上线后默认行为应与当前 Worker 尽量一致。

此表只保存当前全局值，不提供历史版本。

## 管理员 API

新增管理员接口：

- `GET /api/admin/ingest/system-prompts`
  - 返回 `text_expand` 和 `image_reverse` 两条配置及更新时间。
- `PUT /api/admin/ingest/system-prompts/:flowType`
  - 请求体：`{ "prompt": "..." }`。
  - 只接受两个允许的流程键。
  - 验证去除首尾空白后的长度为 1–20,000。
  - 更新 `updated_by` 和 `updated_at`。

现有任务接口增加本次提示词快照处理：

- `POST /api/admin/jobs` 的 `text_expand` 输入可包含 `systemPrompt`。
- `POST /api/admin/jobs/image-reverse` 的 multipart 表单可包含 `systemPrompt`。
- 如果客户端未提供 `systemPrompt`，API 从对应全局配置加载默认值。
- API 验证后，把最终生效值写入 `generation_jobs.input.systemPrompt`，再入队。

API 是快照的权威边界。Worker 不在执行时重新读取全局配置，避免 queued 期间全局默认值变化导致任务行为漂移。

## Worker 数据流

### 提示词优化

```text
文本 + Model + systemPrompt
  → API 校验并写入 generation_jobs.input
  → BullMQ
  → Worker resolvePrimaryModel(text_expand)
  → structurePrompt 使用 input.systemPrompt
  → templateDraftSchema 校验
  → generation_jobs.output + succeeded
```

### 图片反推

```text
图片 + Model + systemPrompt
  → API 上传临时图片并写入任务快照
  → BullMQ
  → Worker resolvePrimaryModel(image_reverse)
  → 若主模型支持 vision：图片 + systemPrompt 直接结构化
  → 否则：默认视觉模型提取图片事实，再由主模型 + systemPrompt 结构化
  → templateDraftSchema 校验
  → generation_jobs.output + succeeded
```

Worker 对部署前创建、没有 `input.systemPrompt` 的遗留任务使用与当前行为等价的内置默认值，避免升级时旧任务直接失败。新任务都应携带 API 解析后的快照。

## 状态与错误处理

- 输入无效时在提交前就地提示。
- queued/running 时禁用本流程重复提交，保留另一流程可用。
- 任务失败时保留输入、模型和系统提示词，显示安全的错误信息并允许调用现有 retry 接口。
- retry 使用原任务已经保存的系统提示词快照。
- 单次轮询网络失败不把业务任务标记为 failed；界面显示连接提示并继续轮询。
- 系统提示词保存失败只影响全局配置更新，不清空当前编辑值。
- 生成结果不满足 `TemplateDraft` 时由 Worker 将任务置为 failed，不向草稿编辑器传入部分或不合法数据。
- 模板保存失败时保留已编辑草稿，允许再次提交。

## 测试与验收

### API

- 非管理员不能读取或更新系统提示词。
- 只接受 `text_expand` 和 `image_reverse`。
- 空值和超过 20,000 字符的值被拒绝。
- 更新一条流程配置不会改变另一条。
- 未提供本次提示词时使用对应全局默认值。
- 提供临时提示词时写入任务快照，但不修改全局配置。
- 全局默认值在任务入队后改变，不影响已保存的任务输入。

### Worker

- `text_expand` 使用文本优化提示词快照。
- `image_reverse` 的直接 vision 路径和视觉 fallback 路径都在最终结构化阶段使用图片反推提示词快照。
- retry 继续使用原快照。
- 缺少快照的遗留任务使用兼容默认值。
- 所有输出继续通过 `templateDraftSchema`。

### Web

- 两个入口卡片能切换聚焦工作区。
- 两套输入、模型、任务和结果状态互不污染。
- 切换入口时另一流程继续轮询，卡片状态同步更新。
- 高级设置的临时修改、恢复默认和保存全局配置反馈正确。
- 同一流程活跃任务期间不能重复提交，另一流程仍可提交。
- 完整草稿字段可编辑，非法草稿不能保存。
- 保存成功携带正确 `source` 和 `sourceMeta.jobId`，并跳转到模板编辑器。

### 集成验收

1. 同时提交提示词优化和图片反推任务。
2. 在两入口之间来回切换，确认输入和进度都保留。
3. 分别得到合法的可编辑草稿。
4. 修改两个草稿并各自保存。
5. 确认创建了两个来源正确的草稿模板。
6. 修改图片反推全局系统提示词，确认不影响提示词优化默认值，也不改变已经 queued 的任务。

## 成功标准

- 用户能清楚区分两类智能入库流程，并在同一页面独立操作。
- 两条异步任务可并行存在且切换不丢状态。
- 每条流程拥有独立、可维护、可临时覆盖的全局系统提示词。
- 所有 AI 结果先经过完整校对，合法后才能保存为草稿模板。
- 现有任务中心、模板编辑器和 Worker 模型路由继续正常工作。

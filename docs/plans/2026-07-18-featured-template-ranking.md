# Promptix 精选模板排序与后台运营实施方案

## 1. 目标

在现有“热门、高赞、最新”排序之间增加“精选”。精选由管理员人工标记，精选模板优先出现，剩余位置由热门模板自动补齐。

该功能必须满足：

- 没有精选模板时，精选结果与热门结果一致。
- 有 N 个已发布精选模板时，前 N 个位置展示精选模板，之后展示未重复的热门模板。
- 搜索、分类、场景、风格和画面主体筛选先收窄候选集合，再在候选集合内执行精选排序。
- 分页、刷新和相同指标下的结果顺序稳定，不出现重复或漏项。
- 草稿和已下架模板即使被标记为精选，也不进入公开列表。

## 2. 当前排序基线

| 入口 | 主排序字段 | 含义 |
| --- | --- | --- |
| 热门 | `use_count DESC` | 使用次数更多的模板优先 |
| 高赞 | `favorite_count DESC` | 收藏/点赞更多的模板优先 |
| 最新 | `created_at DESC` | 创建时间更晚的模板优先 |

所有排序都应追加 `created_at DESC, id ASC` 作为稳定的决胜字段。

## 3. 精选业务规则

### 3.1 数据字段

- `is_featured boolean`：管理员是否将模板设为精选，项目已有该字段。
- `featured_order integer`：精选内部人工顺序，数值越小越靠前，默认 `0`。

取消精选时保留或重置顺序都不应影响热门补位。本实现将非精选模板的顺序值重置为 `0`，降低后台数据歧义。

### 3.2 统一排序表达式

精选模式使用一条数据库查询完成：

```sql
ORDER BY
  is_featured DESC,
  featured_order ASC,
  use_count DESC,
  created_at DESC,
  id ASC
```

这条排序天然完成以下行为：

1. 所有精选模板位于非精选模板之前。
2. 精选模板按人工顺序排列；顺序相同时按热门程度排列。
3. 精选内容结束后，未精选模板继续按热门程度排列。
4. 每条模板只参与一次查询，因此不会重复。
5. 没有精选模板时，`is_featured` 和 `featured_order` 对所有记录相同，结果自然退化为热门排序。

### 3.3 筛选与精选的关系

精选不是绕过筛选的全站固定内容。先应用公开状态、关键词、分类、标签和场景条件，再对匹配结果进行精选排序。例如用户选择“人物肖像与头像”后，只会优先展示该场景中的精选模板。

## 4. API 契约

### 4.1 公开模板列表

`GET /api/templates?sort=featured`

- `sort` 支持 `hot | featured | favorites | latest`。
- 未知值回退到 `hot`，避免旧客户端或错误链接导致接口失败。
- 返回模板包含 `isFeatured` 和 `featuredOrder`，保证静态回退数据与 API 数据能够执行相同的客户端排序。

### 4.2 后台模板列表

`GET /api/admin/templates?featured=true|false`

- 空值：全部模板。
- `true`：只看精选模板。
- `false`：只看非精选模板。

### 4.3 后台编辑

`PATCH /api/admin/templates/:id`

接受：

```json
{
  "isFeatured": true,
  "featuredOrder": 10
}
```

`featuredOrder` 必须是 `0` 到 `1000000` 的整数。

## 5. 前台交互

- 排序顺序固定为：热门、精选、高赞、最新。
- 精选沿用现有排序按钮交互和 URL 参数，地址为 `?sort=featured`。
- 桌面端和移动端使用同一排序枚举。
- 静态数据回退时执行与服务端一致的比较函数，避免 API 不可用时行为变化。

## 6. 后台交互

模板列表：

- 增加“精选状态”筛选：全部、仅精选、非精选。
- 表格显示精选标识和精选顺序。
- 提供快速“设为精选/取消精选”操作。

模板编辑：

- 增加“设为精选”复选框。
- 选中后可填写精选顺序。
- 文案明确“数值越小越靠前；相同时按热门程度排序”。

## 7. 迁移与兼容性

- 数据库迁移只新增 `featured_order`，不删除或改写已有字段。
- 历史模板默认顺序为 `0`，已有 `is_featured=true` 数据继续有效。
- 种子数据同步写入 `featuredOrder`。
- 旧 URL、旧模板记录和未传 `featuredOrder` 的后台请求继续兼容。

## 8. 测试与验收

自动化测试至少覆盖：

- 前台排序枚举包含 `featured`，且入口位于热门和高赞之间。
- 客户端精选比较器满足精选优先、人工顺序、热门补齐和零精选退化。
- API 精选排序使用单条稳定排序表达式。
- 后台输入校验、精选筛选和编辑控件存在。
- 迁移为纯增量，并为字段设置非空默认值。
- Web、共享包和 API 的测试与生产构建通过。

人工验收场景：

1. 无精选模板：精选与热门顺序一致。
2. 设置 1 个精选：该模板位于首位，其余按热门补齐。
3. 设置 10 个精选：前 10 个均为精选，第 11 个开始为非精选热门。
4. 两个精选顺序不同：较小顺序优先。
5. 两个精选顺序相同：使用次数较多者优先。
6. 取消精选：该模板回到热门序列中的自然位置。
7. 精选草稿或下架模板：公开接口不返回它。
8. 翻页：模板不重复且顺序稳定。

# Promptix 模板治理后台管理员操作指南

本指南面向刚接管 Promptix 后台的管理员。目标不是让你理解代码，而是帮助你安全地完成模板巡检、Agent 分析、审批、重试、回滚和规则调整。

## 1. 先理解这套后台在做什么

模板治理不是让 Agent 直接修改所有模板，而是一条受控流程：

```text
选择工作范围 -> Agent 生成建议 -> 系统判定风险 -> 自动执行低风险项
                                      -> 等待管理员审批高风险项
                  -> 查看执行结果 -> 必要时重试或回滚 -> 留下版本与审计记录
```

系统分工如下：

- Agent 负责检查模板并提出结构化建议。
- 规则负责判定建议能否自动执行，Agent 不能自行降低风险等级。
- 管理员负责审批发布、归档、永久删除、Prompt 骨架和变量等高风险变更。
- 系统为非删除变更保存不可变版本和审计记录。

管理入口为 `/admin/templates`，页面名称是“智能分拣台”。

## 2. 第一次接管时先做什么

首次登录后，不要立即批准历史积压或大幅调整规则。建议按下面顺序完成接管检查。

1. 打开“智能分拣台”，确认工作队列、模板列表和右侧检查器都能正常加载。
2. 点开“治理规则”，记录当前定时巡检是否启用、Cron、时区、扫描上限和自动处理阈值。
3. 确认时区符合团队预期。默认值为 `Asia/Shanghai`，默认计划是每天 `03:00`。
4. 查看各队列数量，优先了解“失败与冲突”和“等待审批”是否有历史积压。
5. 随机选择几个模板，核对当前版本、Agent 建议、置信度和版本历史。
6. 第一次人工巡检只选择少量模板，先验证完整流程，再扩大范围。

接管当天建议保存一份交接记录，包括当前规则版本、队列数量、未处理审批、失败项和已知故障。不要通过数据库直接修改治理状态。

## 3. 认识“智能分拣台”

页面分为三个主要区域。

### 左侧：工作队列

左侧用于切换不同治理队列，数字代表当前符合条件的模板数量。选择队列后，中间列表会同步更新。

建议的处理顺序是：

1. 失败与冲突：先确认是否有阻塞性问题。
2. 等待审批：处理已经生成、但需要人工决策的计划。
3. 分类、质量或精选相关队列：开展日常治理。
4. 无待办队列：通常不需要操作，不要为了“清零”而制造变更。

队列数量与列表不一致时，先刷新页面。如果仍不一致，暂停批量操作并交给技术人员检查查询条件和服务状态。

### 中间：模板列表

列表是主要工作区。你可以选择一行查看详情，也可以勾选多行形成明确的批量范围。

选择范围有两种：

- 明确选择：只处理你勾选的模板，适合首次操作和高风险任务。
- 当前筛选结果：处理当前查询命中的全部模板，适合规则成熟后的批量巡检。

使用“当前筛选结果”前必须再次确认筛选条件和预计数量。列表筛选变化后，不要沿用旧范围的判断。

### 右侧：检查器

右侧显示所选模板的当前版本、Agent 建议、风险、置信度、原因、拟修改内容、审批区和版本历史。

审批前至少核对：

- 建议针对的模板是否正确；
- 建议基于的模板版本是否仍是当前版本；
- 修改字段是否符合你的任务目标；
- Agent 的解释是否与实际改动一致；
- 风险等级和置信度是否合理；
- 是否包含 Prompt、变量、发布状态或删除等高风险字段。

置信度高不等于业务判断正确。它只表示 Agent 对结构化建议的确定程度。

## 4. 发起一次 Agent 巡检

### 4.1 先确定范围

1. 在左侧选择目标队列。
2. 使用列表条件缩小范围。
3. 勾选少量模板，或确认要使用当前筛选结果。
4. 记录预计模板数量，避免误操作全量数据。

### 4.2 编写指令

在顶部输入清晰、单一目标的指令。例如：

```text
检查所选模板的分类是否准确，只提出分类和标签调整建议，不修改 Prompt、变量和发布状态。
```

```text
检查当前筛选结果中的低质量描述，补充摘要并标记疑似重复模板，不执行删除。
```

避免使用“全部优化一下”“该改的都改了”这类边界不清的指令。一次任务只处理一个主要目标，更容易审核和回滚。

### 4.3 预览并提交

1. 点击“解释指令”。
2. 核对页面显示的解释范围和预计模板数量。
3. 如果范围不正确，点击“修改”，重新调整筛选或指令。
4. 确认无误后点击“确认提交”。

提交只代表创建治理运行。高风险变更不会因此绕过审批。

## 5. 如何审核 Agent 建议

逐项查看右侧建议，不要只看摘要数字。审批判断可按以下顺序进行。

1. 看当前值和建议值，确认到底改了什么。
2. 看解释和原因代码，判断改动是否有业务依据。
3. 看风险等级和置信度，但不要用它们替代人工判断。
4. 检查建议是否越过本次指令范围。
5. 抽查代表性模板，尤其是批量计划中的边界案例。

下列变更必须经过管理员审批：

- 发布模板；
- 归档模板；
- 永久删除模板；
- 修改 Prompt 骨架；
- 修改模板变量；
- 超出治理规则边界的精选调整。

低风险元数据变更只有在满足当前规则的置信度和批次限制时才可自动执行。

## 6. 批准、拒绝和永久删除

### 批准

1. 确认规则版本、变更范围和代表性差异。
2. 在审批备注中写明批准依据，批量操作尤其要注明抽查范围。
3. 点击“批准”。
4. 等待执行结果，不要因为页面短暂未刷新而重复提交。

### 拒绝

建议填写明确原因，例如“分类依据不足”“超出本次任务范围”或“需内容负责人确认”，然后点击“拒绝”。清晰的拒绝理由有利于后续审计和重新规划。

### 永久删除

永久删除不可回滚，应作为最后手段。

1. 确认模板已经不再需要，且没有保留、归档或修正的价值。
2. 填写具体删除原因。
3. 在确认框中准确输入“永久删除”。
4. 再次核对模板名称和范围，然后批准。

只要存在疑问，先归档，不要永久删除。

## 7. 查看执行结果

变更集摘要会将项目分为：

| 状态 | 含义 | 管理员动作 |
| --- | --- | --- |
| 自动 | 符合规则，可自动执行 | 抽查执行结果 |
| 审批 | 高风险或超出自动规则 | 人工批准或拒绝 |
| 冲突 | 模板或规则在计划后发生变化 | 重新生成计划 |
| 跳过 | 无需变更或不符合执行条件 | 查看原因，通常无需处理 |
| 失败 | 执行时发生可见错误 | 判断是否可重试或升级排障 |

执行结束后应检查：

- 成功、失败、冲突和跳过数量之和是否符合计划范围；
- 抽查后的模板值是否与已批准建议一致；
- 模板版本是否增加；
- 公开模板的展示和排序是否符合预期；
- 是否出现计划外字段变更。

## 8. 失败、冲突和重试

### 部分成功

部分成功表示部分模板已经生效，另一些失败。不要重新执行整个批次，否则可能重复处理已成功项目。

1. 查看失败项及错误原因。
2. 确认可重试项没有被人工再次修改。
3. 点击“重试失败项”，系统只会重试允许重试的项目。
4. 重试后再次核对汇总数量。

### 版本冲突

版本冲突表示计划生成后，模板已经被其他人或任务修改。旧计划不能继续使用。

处理方式：刷新模板 -> 查看最新版本 -> 重新发起 Agent 计划 -> 重新审批。

### 规则版本变化

如果计划引用的规则版本不再是活动版本，系统会要求重新生成计划。不要尝试绕过，这是为了确保审批依据与执行规则一致。

### 常见服务故障

- 页面无法加载或操作返回 401/403：重新登录；仍失败时确认账号角色。
- 任务长期停留在排队：联系技术人员检查 Worker 和 Redis。
- Agent 分析失败：联系技术人员检查默认文本模型及结构化输出能力。
- 所有写操作失败：暂停审批，联系技术人员检查数据库和迁移状态。

## 9. 如何回滚

回滚不是删除历史，而是创建一个新的前向版本，将内容恢复到变更前状态。

允许回滚通常需要同时满足：

- 变更不是永久删除；
- 当前时间未超过回滚期限；
- 模板在该次变更后没有产生新的修改；
- 当前版本仍与原执行结果一致。

操作步骤：

1. 打开对应变更集。
2. 确认错误来自该次变更，而不是之后的人工修改。
3. 查看回滚期限。
4. 点击“回滚变更”。
5. 回滚完成后抽查模板内容、版本和公开展示。

如果显示“回滚期限已过”或版本冲突，不要直接修改数据库。应基于当前模板生成一份新的修正计划。

## 10. 治理规则怎么改

从“智能分拣台”右上角打开“治理规则”。每次保存都会创建新版本，不会改写历史运行引用的规则。

| 配置 | 作用 | 调整建议 |
| --- | --- | --- |
| 启用定时巡检 | 控制是否自动发起每日巡检 | 故障或维护期间可暂时关闭 |
| Cron | 设置巡检时间 | 避开发布和内容编辑高峰 |
| 时区 | 解释 Cron 的时区 | 通常保持 `Asia/Shanghai` |
| 扫描上限 | 单次最多检查多少模板 | 新接管时保持较小值 |
| 最低自动置信度 | 低于此值不得自动执行 | 降低会扩大自动变更范围 |
| 自动批次上限 | 单批自动执行数量上限 | 应与团队复核能力匹配 |
| 回滚小时 | 自动变更的可回滚窗口 | 不宜短于日常复核周期 |
| 精选槽位 | 精选模板目标数量 | 与前台实际展示位匹配 |
| 最大替换比例 | 单次可自动替换的精选比例 | 越低越保守 |
| 精选冷却小时 | 两次精选调整的最短间隔 | 防止前台频繁波动 |

安全修改流程：

1. 记录当前值和修改原因。
2. 一次只调整一类参数。
3. 先缩小扫描和自动批次范围进行验证。
4. 点击“保存为新版本”。
5. 观察下一次小批量运行及前台效果。
6. 确认稳定后再逐步扩大范围。

不要为了快速清理积压而同时降低置信度、提高扫描上限和提高自动批次上限。

## 11. 推荐的日常工作节奏

### 每天

1. 查看失败与冲突。
2. 查看等待审批项目。
3. 抽查最近自动执行结果。
4. 确认没有任务异常长期停留在排队或执行中。

### 每周

1. 复盘审批通过率、拒绝原因和失败原因。
2. 抽查模板分类、Prompt、变量和精选展示质量。
3. 检查自动规则是否过严或过松。
4. 检查回滚次数和版本冲突是否异常增加。

### 调整规则或升级模型后

1. 使用小范围明确选择发起测试运行。
2. 人工审核全部建议。
3. 只批准低影响批次。
4. 对比变更前后结果。
5. 稳定后再恢复常规扫描上限。

## 12. 一次完整操作示例

目标：检查近期模板的分类，只自动修正高置信度元数据，不修改 Prompt 或发布状态。

1. 进入 `/admin/templates`。
2. 选择分类相关队列，并将范围缩小到近期模板。
3. 明确勾选 5 至 10 个模板。
4. 输入：“检查所选模板分类，只建议分类和标签调整，不修改 Prompt、变量、精选和生命周期。”
5. 点击“解释指令”，确认范围与数量。
6. 点击“确认提交”。
7. 在右侧逐项检查当前值、建议值、解释、风险和置信度。
8. 等待符合规则的低风险项自动执行。
9. 对需要审批的意外高风险项选择拒绝，并注明“超出任务范围”。
10. 检查执行摘要，抽查成功模板的新版本和前台效果。
11. 如果某项结果错误且仍在期限内，执行回滚；否则创建新的修正计划。

## 13. 什么时候必须停止操作并升级处理

出现以下情况时，暂停批量审批并联系技术负责人：

- 队列数量与列表持续不一致；
- Agent 建议包含本次指令禁止修改的字段；
- 自动执行突破配置的置信度或批次限制；
- 发布、归档、删除、Prompt 或变量变更未经过审批；
- 多个任务持续排队、失败或反复重试；
- 版本历史缺失、不连续或与当前版本不符；
- 活动治理规则不是唯一一份；
- 回滚后数据或前台展示没有恢复；
- 怀疑发生越权操作或审计记录缺失。

技术排障、服务恢复、监控指标和 API 检查方法见 [模板治理运维手册](../superpowers/runbooks/template-governance-operations.md)。
